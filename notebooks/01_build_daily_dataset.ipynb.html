Daily Occupancy Forecasting & Demand Dynamics Modeling¶

Reconstructing Daily Occupancy Dynamics from Weekly Stay Records¶

Business Context¶

This project is based on real operational data from a medium-term accommodation business, where units are rented for extended stays rather than short hotel-style visits.

The core business problem was limited visibility into short- and medium-term demand. Occupancy could shift unexpectedly, especially in lower-demand periods, making it difficult to support pricing, availability, and promotional decisions with confidence.

The aim of this project is to build a dataset that allows the business to:

  • understand daily occupancy dynamics
  • detect and anticipate demand increases and drops
  • support forecasting and operational decision-making

Unlike a standard hotel setting, demand here is shaped not only by seasonality, but also by booking timing, notices or cancellations, stay duration, and operational constraints such as available units.

Operational Logic of the Business¶

This business follows a continuous occupancy model:

  • Customers remain in the system until notice is given
  • Departures are not tied to fixed checkout cycles
  • Occupancy evolves through inflows and outflows

This can be expressed as a state-transition process: Occupancy_{t+1} = Occupancy_t + Bookings − Notices

Where:

  • Occupancy_t is the current system state
  • Bookings represent inflows (future occupancy increases)
  • Notices represent outflows (future occupancy decreases)
  • Timing delays depend on booking lead time and notice behavior

Timing also matters: bookings made far in advance and notices given close to departure do not affect the system in the same way.

Why this matters¶

This is not just a generic demand-forecasting problem. It is better understood as a state evolution problem, where future occupancy depends on the interaction between:

  • current occupancy
  • incoming demand
  • outgoing demand
  • timing behavior

That operational logic shapes the full project, starting with the dataset design.

Original Dataset Structure¶

The source data is organized as a weekly tracking structure.

Each row represents: 1 Customer × 1 Unit × 1 Week

Because of this, a single stay appears across multiple rows rather than as one continuous record.

Key limitation¶

This structure introduces important issues:

  • Booking and notice events are repeated across weeks
  • Temporal precision is limited to weekly granularity
  • Demand spikes and short-term dynamics are smoothed or diluted

As a result, the raw structure is not suitable for daily forecasting or operational signal extraction.

Transformation Objective¶

The goal of Notebook 1 is to convert the weekly tracking data into a clean daily time series that reflects the real operating process.

This daily structure allows:

  • accurate reconstruction of daily occupancy
  • correct counting of booking and notice events
  • preservation of timing information
  • analysis of short-term demand dynamics

This transformed dataset becomes the analytical foundation for all later modeling and decision-support steps.

Key Variables¶

Stay Structure¶
  • Customer_ID, Unit_ID
  • MoveIn_Date, MoveOut_Date
  • Week
  • Occupied_Days (number of days occupied within that week)
Booking & Notice Behavior¶
  • Booking_Date, Booking_Time
  • Notice_Date, Notice_Time
  • Cancelled
Customer & Stay Attributes¶
  • Age, Sex
  • With_Agent, Type_Pref
  • Stay_Length, PreArrival_CheckOut

2. Data cleaning and stay-level reconstruction¶

The original dataset is structured as weekly snapshots, where each stay appears multiple times. To recover the true operational timeline, stays are first deduplicated using unique identifiers and validated for date consistency.

Each valid stay is then expanded into a daily sequence of records, transforming the data from a static weekly representation into a continuous time series. This enables accurate tracking of unit occupancy and preserves the temporal structure of customer stays.

3. Build daily booking and notice signals from deduplicated stays¶

Booking and notice events are aggregated at the daily level using the deduplicated stay dataset. This ensures that each stay contributes exactly one booking and one notice event, avoiding distortions caused by repeated weekly records.

The resulting signals represent:

  • daily inflows (bookings)
  • daily outflows (notices)
  • associated timing behavior (average booking and notice lead times)

These variables provide a behavioral view of demand dynamics aligned with the business process.

4. Merge daily occupancy with event signals¶

The occupancy table is combined with the daily booking and notice signals into a single dataset indexed by date.

Additional indicator variables distinguish between:

  • days where no event occurred
  • days where timing values are missing because there was no event

This is important because “no booking happened” is not the same as “booking timing is missing.”

5. Build a complete daily calendar¶

A continuous daily calendar is then created to ensure there are no gaps in the time series. This step:

  • fills missing dates
  • preserves a missing-data flag for occupancy where relevant
  • re-aligns capacity by week
  • recomputes occupancy on the final daily calendar

The result is a complete daily dataset ready for forecasting.

In [11]:
daily["Date"] = pd.to_datetime(daily["Date"])

full_calendar = pd.DataFrame({
    "Date": pd.date_range(daily["Date"].min(), daily["Date"].max(), freq="D")
})

daily = full_calendar.merge(daily, on="Date", how="left")
daily = daily.sort_values("Date").reset_index(drop=True)

# Rebuild ISO week after calendar expansion
daily["Week"] = daily["Date"].dt.strftime("%G-W%V")

# Re-merge weekly units_available to ensure all days get a denominator
daily = daily.drop(columns=["units_available"], errors="ignore")
daily = daily.merge(weekly_units, on="Week", how="left")

# Do NOT assume missing = 0 immediately
daily["Occupied_Units"] = daily["Occupied_Units"]
# Keep track of missing days
daily["Missing_Occupancy_Flag"] = daily["Occupied_Units"].isna().astype(int)
# Forward fill ONLY small gaps
daily["Occupied_Units"] = daily["Occupied_Units"].fillna(method="ffill")
# If still NaN at the very beginning, backfill
daily["Occupied_Units"] = daily["Occupied_Units"].fillna(method="bfill")

# Fill missing activity values
daily["Bookings"] = daily["Bookings"].fillna(0)
daily["Notices"] = daily["Notices"].fillna(0)
daily["Has_Bookings"] = daily["Has_Bookings"].fillna(0).astype(int)
daily["Has_Notices"] = daily["Has_Notices"].fillna(0).astype(int)

daily["Avg_Booking_Time"] = daily["Avg_Booking_Time"].fillna(daily["Avg_Booking_Time"].median())
daily["Avg_Notice_Time"] = daily["Avg_Notice_Time"].fillna(daily["Avg_Notice_Time"].median())

# Recompute occupancy after full calendar + denominator merge
daily["Occupancy"] = daily["Occupied_Units"] / daily["units_available"]

# Filter dataset to valid period (when units_available exists)
cutoff_date = pd.Timestamp("2016-08-01")

daily = daily[daily["Date"] >= cutoff_date].copy()
